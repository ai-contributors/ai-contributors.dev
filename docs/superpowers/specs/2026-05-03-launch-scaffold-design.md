# AI Contributors Dev Launch Scaffold Design

## Purpose

Build the first working scaffold for `ai-contributors.dev`: an Astro + Starlight documentation site that renders the AI Contributor Specification from the `ai-contributors/ai-contributor-spec` Git submodule without committing copied specification text into this repository.

This design covers the launch scaffold only. It intentionally excludes the full conformance automation pass: scheduled spec bump pull requests, repository dispatch handling, weekly audit workflow, PR deploy previews, dependency/security scanning, committed audit artifacts, DNS validation, and branch protection evidence.

## Goals

- Create a working docs site that can build locally and in GitHub Actions.
- Source formal specification content from `external/ai-contributor-spec/`.
- Keep generated spec docs out of git so this repository does not duplicate normative content.
- Provide the docs-first homepage direction selected during brainstorming.
- Provide a conformance badge generator for paste-ready Markdown badge snippets.
- Fail the build loudly when the submodule or expected source files are missing.
- Establish enough CI and Pages deployment structure for later automation work to build on.

## Non-Goals

- No web-based audit runner.
- No blog, leaderboard, or third-party submission flow.
- No scheduled submodule update PR workflow in this first plan.
- No weekly `ai-contributor-audit` workflow in this first plan.
- No committed audit result or L3 conformance claim enforcement in this first plan.
- No custom design system beyond restrained Starlight styling and the existing logo assets.

## Architecture

The site uses Astro with Starlight. Source Markdown from the spec submodule is treated as an external input. A Node script validates the source manifest, reads the submodule version metadata, and generates ignored Starlight content files before Astro runs.

The generated docs files live under a gitignored generated content directory, for example `src/content/docs/_generated/`. They are created from the pinned submodule on each build and are not manually edited. Site-authored pages, such as the homepage and short audit/skills bridge pages, stay in committed source files and avoid normative specification text.

This approach keeps Starlight routing conventional while preserving the source-of-truth rule: committed repository files do not contain copied specification content. If a future Astro/Starlight version makes external content loaders easier for docs routes, the generator can be replaced without changing public routes.

## Source Inputs

The scaffold expects the spec submodule at:

- `external/ai-contributor-spec/`

The manifest check covers these source files:

- `AI-CONTRIBUTOR-SPECIFICATION.md`
- `AI-CONTRIBUTOR-AUDIT-MODEL.md`
- `AI-CONTRIBUTOR-AUDIT-PROMPT.md`
- `AI-CONTRIBUTOR-GUIDE.md`
- `AI-CONTRIBUTOR-COVERAGE.md`
- `CHANGELOG.md`
- `skills/ai-contributor-audit/README.md`
- `skills/ai-contributor-audit/SKILL.md`
- `skills/ai-contributor-audit-fix/SKILL.md`
- `skills/ai-contributor-audit-profile/SKILL.md`

If the submodule is missing, uninitialized, or any file is absent, `pnpm build` fails before Astro renders the site.

- `/levels/` from a committed page using the badge generator and linked conformance level context.
- `/audit/` from a committed bridge page.
- `/audit/model/` generated from `AI-CONTRIBUTOR-AUDIT-MODEL.md`.
- `/audit/prompt/` generated from `AI-CONTRIBUTOR-AUDIT-PROMPT.md`.
- `/skills/` from a committed overview page.
- `/skills/audit/` generated from `skills/ai-contributor-audit/README.md`.
- `/skills/audit/skill/` generated from `skills/ai-contributor-audit/SKILL.md`.
- `/skills/audit-fix/` generated from `skills/ai-contributor-audit-fix/SKILL.md`.
- `/skills/audit-profile/` generated from `skills/ai-contributor-audit-profile/SKILL.md`.

If the llms plugin is not compatible during implementation, the scaffold should provide a small static generation script for those two files rather than delaying the site scaffold.

## Homepage

The homepage uses the docs-first landing direction selected in brainstorming.

It includes:

- Starlight navigation with links to the spec, audit, skills, levels, and GitHub.
- A short hero using the specification repository README lead copy where available.
- Primary CTAs: `Read the spec`, `Run the audit`, and `Get a badge`.
- A compact L0-L4 `LevelLadder` component.
- An `InstallCommand` component with a copyable install command.
- A short non-normative "Why this exists" section, sourced or paraphrased conservatively from the spec README during implementation.
- A seven-pillar card grid linking into the specification page.
- The repository's current claimed conformance badge slot, initially configured for Level 3 per the website specification.
- A source footer showing the pinned spec tag and short SHA when available.

The homepage remains fully static and performs no runtime GitHub API calls.

## Components

The scaffold uses small Astro components with clear responsibilities:

- `LevelLadder.astro`: renders L0-L4 as compact linked steps.
- `BadgeGenerator.astro`: renders the level selector and paste-ready Markdown output.
- `InstallCommand.astro`: renders the canonical install command and copy affordance.
- `SpecSourceFooter.astro`: renders the source version tag and short SHA derived from the submodule.
- `Hero.astro`: keeps homepage hero markup separate from page content.

Client-side JavaScript is limited to the badge generator and copy buttons. All docs content remains static.

## Generated Content

The generator maps each external source file to a Starlight docs route. It adds frontmatter needed by Starlight, then appends the external Markdown body unchanged except for mechanical compatibility changes required for routing or rendering.

The generator must not rewrite normative wording. If a source file needs substantial transformation to render well, that transformation should be isolated, documented in code, and covered by tests.

Generated files are deleted and recreated on each run so stale routes cannot survive after source changes.

## Version Metadata

The scaffold derives source metadata from the submodule:

- short SHA: `git -C external/ai-contributor-spec rev-parse --short HEAD`
- exact SHA: `git -C external/ai-contributor-spec rev-parse HEAD`
- release tag: the exact tag pointing at `HEAD`, falling back to the nearest tag when no exact tag is available

Footer output uses `v0.1 (<short-sha>)` style when the launch submodule is pinned to `v0.1`. The tag links to the upstream tag or release page; the SHA links to the exact tree.

## Tooling

The implementation should use:

- Node 24.x.
- Pinned `pnpm` via `packageManager`.
- Astro and Starlight pinned to exact versions.
- TypeScript through Astro's default configuration.
- A committed lockfile.

Dependency versions should be chosen at implementation time from current stable releases and pinned exactly. Astro/Starlight compatibility should be established with the official upgrade/scaffold tooling where practical.

## Testing

The scaffold should add focused tests for build-critical behavior:

- The source manifest validator succeeds when all expected files exist.
- The source manifest validator fails with a clear error when the submodule path is missing.
- The docs generator creates the expected route files from the external source map.
- Generated route files are not committed source-of-truth files.
- Badge generator data produces the expected shields.io Markdown snippets for all five levels.

These tests can run with Node's built-in test runner or a minimal pinned test runner. The implementation should choose the smallest setup that fits Astro and TypeScript cleanly.

## CI And Deployment

The scaffold adds a basic GitHub Actions workflow for build and Pages deployment.

On pull requests and pushes:

- Check out the repository with recursive submodules.
- Set up Node 24.
- Enable the pinned pnpm version.
- Run `pnpm install --frozen-lockfile`.
- Run the source manifest validation.
- Run tests.
- Run `pnpm check`.
- Run `pnpm build`.

On `main` only:

- Upload the `dist/` artifact for GitHub Pages.
- Deploy through `actions/deploy-pages`.

The workflow does not create PR previews or scheduled update PRs in this scaffold pass.

## Repository Hygiene

The implementation should add `.superpowers/` to `.gitignore` so brainstorming companion artifacts do not appear as normal untracked work.

Generated docs should also be gitignored. The repository should contain source scripts and route mapping, not generated copies of the specification.

The existing untracked `docs/design-logos/` assets are treated as user-provided design inputs. The implementation can use them for favicon and Open Graph assets when building the scaffold, but should not remove or rewrite them.

## Risks And Mitigations

- Starlight may not support external docs sources directly. The generated ignored docs approach avoids this by using Starlight's normal content layout.
- Generated Markdown transformations could accidentally alter normative content. Keep transformations mechanical, small, and covered by tests.
- The spec submodule may not be present during early local work. The manifest check should fail with setup instructions, not a vague file-not-found stack trace.
- Some plugins may lag current Astro versions. If `starlight-llms-txt` is incompatible, generate `/llms.txt` and `/llms-full.txt` with a small local script for the scaffold.
- Full self-conformance requires more repository governance than this pass covers. Keep clear extension points for the follow-up conformance automation plan.

## Acceptance Criteria For This Scaffold

- `pnpm build` produces a Starlight site from submodule source content.
- The expected public routes exist in the generated site.
- Missing submodule files fail the build before Astro rendering.
- The homepage is docs-first and fully static.
- Badge generator output covers L0-L4.
- Footer source metadata shows the pinned spec version and short SHA when the submodule is initialized.
- CI installs, validates, tests, checks, and builds the site.
- The generated spec content directory and `.superpowers/` are ignored by git.
